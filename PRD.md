## 1. 개요

### 1.1 배경
사내 공용 차량(업무용 차량)을 부서별로 나눠 관리하고 있으나, 현재는 `carlist.xlsx` 엑셀 파일로 차량 목록과 대여 이력을 수기 관리하고 있다. 예약 충돌 확인, 실시간 상태 파악(운행가능/운행중/정비중), 반납 처리가 모두 수작업으로 이뤄져 비효율과 오류(중복 예약, 반납 누락 등)가 발생하기 쉽다.

### 1.2 목적
엑셀 관리 방식을 대체하는 웹 기반 차량운영시스템을 구축하여, 임직원이 직접 차량 현황을 조회하고 예약/반납을 처리할 수 있도록 한다. 디자인은 **미니멀하면서 세련된(minimal & sophisticated)** 톤으로, 실무에서 매일 열어보는 도구인 만큼 군더더기 없이 빠르게 정보를 파악할 수 있어야 한다.

### 1.3 참고 데이터 (carlist.xlsx 분석 결과)
기존 엑셀은 2개 시트로 구성되어 있으며, 이를 시스템의 핵심 데이터 모델 기준으로 삼는다.

- **차량목록** 시트: 차량 20대 (세단/SUV/승합/트럭/전기차 혼합, 6개 부서 관리)
- **대여대장** 시트: 예약/대여 이력 30건 (예약번호 `R2026###` 형식)

| 차량 상태 실시간 파악 | 상태 확인을 위한 유선/메신저 문의 감소 |

| 엑셀 대체 | 전 부서 실사용 전환, 엑셀 파일 업데이트 중단 |

## 3. 사용자 정의

| 구분 | 설명 | 주요 니즈 |
|---|---|---|
| **일반 임직원 (예약자)** | 전 부서 직원. 업무용 차량을 예약/사용하는 주 사용자 | 이용 가능한 차량을 빠르게 찾고, 예약·반납을 스스로 처리 |
| **관리자 (총무/경영지원팀)** | 차량 등록·정비 상태 관리, 예약 현황 총괄 | 차량 마스터 데이터 관리, 전체 예약 모니터링, 강제 상태 변경(정비 등록 등) |

> 가정: 별도 부서장 승인 워크플로우는 1차 버전 범위에서 제외(선착순 예약 방식). 필요 시 향후 확장.

## 4. 범위

### In Scope (1차 릴리즈)
- 차량 목록 조회, 검색, 필터링
- 차량 상세 정보 확인
- 차량 예약(대여 신청) / 예약 취소
- 대여 시작(출발) / 반납 처리
- 나의 예약 내역 확인
- 관리자용 차량 등록·수정·삭제(CRUD)
- 관리자용 전체 예약 현황 및 상태 강제 변경(정비 등록 등)
- 캘린더/타임라인 형태의 차량별 예약 현황 뷰

### Out of Scope (향후 검토)
- 부서장 승인 결재 라인
- 유류비/정산 관리
- 모바일 앱(네이티브)
- 차량 위치 GPS 연동
- 사고/과태료 이력 관리
- 알림톡/이메일 자동 발송 (1차는 화면 내 상태 표시로 대체)

## 5. 데이터 모델

기존 엑셀 두 시트를 그대로 정규화한 2개 핵심 엔티티로 설계한다.

### 5.1 차량 (vehicles)

| 필드 (엑셀 컬럼) | 컬럼명(제안) | 타입 | 비고 |
|---|---|---|---|
| 차량번호 | `plate_number` | text, PK/Unique | 예: `12가 3456` |
| 차종 | `model` | text | 예: 아반떼, 쏘나타, 그랜저, 카니발, 스타렉스, 포터2, K5, 아이오닉5, 투싼, 쏘렌토, 레이, 모닝, 스포티지, K8, 팰리세이드, 니로, EV6, 그랜드 스타렉스, 봉고3, 제네시스 G80 |
| 연식 | `year` | int | 예: 2019~2023 |
| 색상 | `color` | text | |
| 정원 | `capacity` | int | 3~12인승 |
| 연료 | `fuel_type` | enum | 가솔린 / 디젤 / 하이브리드 / 전기 |
| 부서(관리) | `managing_dept` | text/FK | 총무팀, 개발팀, 마케팅팀, 인사팀, 경영지원팀, 영업팀 |
| 상태 | `status` | enum | 운행가능 / 운행중 / 정비중 |
| 주차위치 | `parking_location` | text | 예: 본관 지하1층 2번 |
| 비고 | `note` | text, nullable | 예: "겨울용 스노우체인 비치" |

### 5.2 예약/대여 (reservations)

| 필드 (엑셀 컬럼) | 컬럼명(제안) | 타입 | 비고 |
|---|---|---|---|
| 예약번호 | `reservation_id` | text, PK | 예: `R2026091` — 시스템 도입 후 자동 채번(UUID 또는 시퀀스)으로 대체 |
| 차량번호 | `vehicle_id` | FK → vehicles | |
| 예약자 | `requester_name` | text | 로그인 연동 시 `user_id`로 대체 |
| 부서 | `dept` | text | 예약자 소속 부서 (차량 관리부서와 다를 수 있음) |
| 대여목적 | `purpose` | text | 예: 거래처 미팅, 공항 픽업, 자재 운반, 행사 지원 등 |
| 출발일시 | `start_at` | datetime | |
| 반납예정일시 | `expected_end_at` | datetime | |
| 실제반납일시 | `actual_end_at` | datetime, nullable | 반납 처리 시 기록 |
| 상태 | `status` | enum | 예약됨 / 대여중 / 반납완료 (+ 취소 상태 신규 추가 권장) |

### 5.3 상태 전이 규칙

**예약 상태**: `예약됨` → (출발 처리) → `대여중` → (반납 처리) → `반납완료`
              `예약됨` → (취소) → `취소됨` *(신규)*

**차량 상태**는 예약 상태와 연동되어 파생 관리:
- 해당 차량에 `대여중` 예약이 있으면 차량 상태 = `운행중`
- 반납 완료 시 차량 상태 = `운행가능` (단, 관리자가 `정비중`으로 수동 전환한 경우는 예외)
- 관리자가 직접 `정비중`으로 전환 가능 (예약 가능 목록에서 제외)

### 5.4 핵심 제약조건
- 동일 차량에 대해 시간대가 겹치는 예약은 생성 불가 (예약 충돌 방지 — 엑셀 관리의 가장 큰 문제점)
- `정비중` 차량은 예약 생성 불가
- `expected_end_at` > `start_at`

## 6. 핵심 기능 요구사항

### 6.1 대시보드 (홈)
- 전체 차량 대수, 상태별(운행가능/운행중/정비중) 요약 카운트 카드
- 오늘 출발/반납 예정인 예약 리스트
- 빠른 예약 진입 CTA

### 6.2 차량 목록
- 카드형 또는 테이블형 리스트 (차종, 차량번호, 부서, 정원, 연료, 상태 뱃지 표시)
- 필터: 상태, 부서, 연료, 정원(인원수), 연식
- 검색: 차량번호, 차종
- 정렬: 차량번호순, 상태순

### 6.3 차량 상세
- 차량 기본 정보 전체 표시
- 해당 차량의 예약 현황(캘린더/타임라인)
- "예약하기" CTA (운행가능 상태일 때만 활성화)

### 6.4 예약(대여 신청)
- 차량 선택 → 예약자명/부서, 대여목적, 출발일시, 반납예정일시 입력
- 실시간 시간대 충돌 검증 (겹치는 예약 존재 시 에러 및 대안 시간 안내)
- 예약 완료 시 상태 `예약됨`으로 등록

### 6.5 출발/반납 처리
- 예약자가 실제 차량을 가져갈 때 "출발 처리" → 상태 `대여중` 전환
- 사용 종료 후 "반납 처리" → `actual_end_at` 기록, 상태 `반납완료` 전환, 차량 상태 자동 `운행가능` 복귀
- 예정 반납 시간을 초과한 `대여중` 건은 지연 표시(뱃지 강조)

### 6.6 나의 예약 내역
- 예약자 기준 예약 이력 목록 (예정/진행중/완료/취소 필터)
- 예정된 예약 취소 기능

### 6.7 관리자 기능
- 차량 등록/수정/삭제 (CRUD)
- 차량 상태 수동 변경 (정비 등록/해제)
- 전체 예약 목록 조회 및 검색 (부서/기간/상태 필터)
- 예약 강제 취소/수정 (관리자 개입이 필요한 경우)

### 6.8 캘린더/타임라인 뷰
- 차량을 행(row), 시간을 축으로 하는 간트 형태의 타임라인으로 예약 현황 시각화
- 특정 기간(주/월) 내 전체 차량 가동률 파악

## 8. 디자인 방향 (미니멀 & 세련됨)

### 8.1 원칙
- **여백 우선**: 정보 밀도를 낮추고 여백으로 위계를 표현. 불필요한 테두리/그림자 최소화
- **절제된 색상**: 뉴트럴(그레이/화이트/블랙) 베이스 + 상태 표시용 포인트 컬러 1~2개만 사용
  - 운행가능: 그린 계열
  - 운행중: 블루/앰버 계열
  - 정비중: 레드/그레이 계열
- **타이포그래피 중심 위계**: 색보다 폰트 크기·굵기·자간으로 정보 구조를 표현
- **일관된 컴포넌트**: 뱃지, 카드, 버튼 등 반복 요소는 하나의 스타일 시스템으로 통일
- **모션은 절제**: 상태 전환 시 미묘한 트랜지션(150-200ms) 정도만 사용, 화려한 애니메이션 지양

### 8.2 컬러 팔레트 (제안)
| 용도 | 색상 |
|---|---|
| Background | `#FAFAFA` (Light) / `#0F1115` (Dark 대응 고려) |
| Surface(카드) | `#FFFFFF` |
| Text Primary | `#171717` |
| Text Secondary | `#6B7280` |
| Border | `#E5E7EB` |
| Accent (Primary CTA) | `#111827` (블랙 계열, 포인트 최소화) |
| 상태-운행가능 | `#16A34A` |
| 상태-운행중 | `#F59E0B` |

### 8.3 타이포그래피
- 시스템 폰트 스택 또는 Pretendard(한글 가독성 우수) 권장
- 헤딩: Semibold, 본문: Regular~Medium, 뱃지/라벨: Medium + 소문자 대비 강조

### 8.4 레이아웃
- 좌측 최소 사이드바(대시보드/차량/예약/관리) + 우측 콘텐츠 영역
- 데스크톱 우선이나 모바일에서도 목록/예약 폼이 무리 없이 동작해야 함 (반응형)

## 10. 기술 스택 (제안)

> 부트캠프 프로젝트 환경 기준 제안이며, 확정 아님.

- **프런트엔드**: HTML / CSS / Vanilla JavaScript 기반 단일 페이지 웹앱 (SPA)
- **백엔드/DB**: Supabase (PostgreSQL + Auth + Realtime)
  - `vehicles`, `reservations` 테이블을 5장 데이터 모델대로 구성
  - Supabase Auth로 로그인, RLS(Row Level Security)로 일반 사용자/관리자 권한 분리
  - Realtime 구독으로 차량 상태·예약 현황 실시간 갱신
- 상세 테이블 스키마 및 정책은 별도 `DB.md`로 정리 후 개발 착수

## 11. 오픈 이슈 / 확인 필요 사항

1. 로그인 방식: 사내 SSO 연동 여부, 또는 간단한 이메일/사번 로그인으로 충분한지
2. 예약 취소 정책: 취소 가능 시점(출발 몇 시간 전까지) 제한 여부
3. 반납 지연/미반납 시 별도 알림·에스컬레이션 필요 여부 (1차 범위 외로 가정)
4. 부서 마스터 데이터(6개 부서) 고정 여부 — 향후 조직 변경 시 관리자가 추가/수정 가능해야 하는지
5. 예약번호 채번 규칙 — 기존 `R2026###` 형식 유지 여부 또는 UUID 전환
